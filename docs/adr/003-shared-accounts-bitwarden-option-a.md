# ADR-003: 공용 계정 정책 및 Bitwarden 옵션 A 구조

- 작성일: 2026-04-22
- 작성자: 장수지, 박성훈
- 상태: accepted

## 배경

엔디즈는 여러 외부 서비스(Firebase, Jira, Notion, Figma, GitHub Organization 등)를 팀 단위로 사용한다. 각 서비스를 어떤 계정으로 가입하고, 비밀번호를 어떻게 공유할지 일관된 원칙이 필요했다.

문제 맥락:
- 개인 Gmail로 Firebase 프로젝트를 만들면 그 사람이 떠날 때 소유권 이관이 번거롭거나 불가능
- 비용 청구가 여러 개인 카드에 흩어지면 추적이 어렵다
- 두 멤버 모두 휴가·장애 상황에서 동등하게 접근 가능해야 한다
- 향후 법인 전환 시 공용 계정이 법인으로 일괄 승계하기 쉽다
- 비밀번호를 카톡·Slack·노션 평문에 두는 것은 유출 위험

비밀번호 관리자(Bitwarden) 사용 방식도 두 가지 선택지가 있었다:
- 옵션 A: 각자 개인 Bitwarden 계정 + 조직(ANDIZ Organization) 공유 Collection
- 옵션 B: 공용 Bitwarden 계정 하나를 두 명이 공유

## 결정

### 공용 계정 우선 정책

모든 외부 서비스는 `andiz`로 시작하는 공용 계정으로 가입한다. 공용 Gmail은 `contact.andiz@gmail.com`.

예외로 두는 개인 계정:
- **Apple Developer**: 현재 Individual 프로그램 특성상 개인 Apple ID 사용 (수지 명의). 법인 전환 시 Organization으로 재가입 예정.
- **GitHub 개인 계정**: 각자 유지, GitHub Organization(`ANDIZ-team`)에 멤버로 초대.

### Bitwarden은 옵션 A 구조

- 각 멤버가 **본인 개인 이메일로 Bitwarden 계정** 가입
- `ANDIZ` Organization을 만들어 두 멤버가 소속
- 공유 대상은 `ANDIZ` Organization의 공유 Collection에만 저장
- 개인 비번은 각자의 "내 보관함"에 저장

### 공유 대상 저장 규칙

- **자체 비번이 있는 서비스**: Bitwarden `ANDIZ 공용 계정` Collection에 Login 타입으로 저장
- **Google SSO 서비스**(Firebase, Jira, Notion, Figma 등): 별도 저장 불필요. 공용 Gmail 항목의 URI 목록에 서비스 URL만 추가
- **2FA 백업 코드**: `ANDIZ 공용 계정` Collection에 Note 타입으로 저장
- **TOTP**: 무료 플랜 미지원. 각자 개인 인증앱(Google Authenticator 등)에서 공용 계정 2FA를 양쪽 폰에 등록

## 근거

### 공용 계정 우선 이유

- **서비스 락인 방지** — 개인 계정 소유 자산은 그 사람 이탈 시 이관이 어렵거나 불가능하다. Firebase 프로젝트 owner 이관은 가능하지만 번거로우며, 멤버 교체마다 반복되는 오버헤드는 주간 출시 모델에 부담이다.
- **비용 청구 일원화** — 모든 구독·결제가 공용 카드(발급 예정) 한 곳으로 모이면 월 고정비 추적이 단순해진다.
- **2명 동등 접근** — 누가 휴가여도 출시·장애 대응 가능.
- **법인 전환 대비** — 법인화 시점에 공용 계정 한 묶음을 법인 자산으로 승계하는 것이, 개인 소유 자산을 개별 이관하는 것보다 훨씬 단순하다.

### Bitwarden 옵션 A 이유

- **Organization 기능 활용** — 옵션 A에서만 "공용 자산"과 "개인 자산"이 한 볼트 안에서 분리된다. 멤버가 개인 비번을 같은 앱으로 관리할 수 있어 Bitwarden 채택률 자체가 높아진다.
- **멤버 교체 내구성** — 옵션 A는 멤버 한 명이 이탈해도 나머지는 자기 계정·자기 볼트를 유지한다. 옵션 B는 한 명이 나갈 때마다 공용 마스터 비번을 교체해야 하고, 그 과정에 실수 여지가 있다.
- **무료 플랜 내 수용 가능** — Bitwarden Free Organization은 2명·Collection 2개까지 무료. 현재 엔디즈 규모와 일치.

### 폐기한 대안

- **Bitwarden 옵션 B(공용 계정 하나 공유)**: 설정은 간단하지만 Organization 기능 의미가 사라지고, 로그인 세션 충돌·마스터 비번 관리 부담·퇴사 시 일괄 변경 부담이 누적된다.
- **1Password 채택**: 상업 도구로 UI·기능은 우수하지만 연 $7.99/user 비용이 들고, 엔디즈 초기 단계엔 Bitwarden 무료로 충분.
- **평문 공유 (카톡/Slack/노션 등)**: 유출 리스크가 크고 변경 시 재전송 부담 누적. 원천 차단.

## 결과

### 얻는 것

- 멤버 이탈·추가 시에도 계정 자산의 소유권이 조직에 남아 이관 비용 최소화.
- 비용이 공용 카드로 집중되어 월별 고정비 가시성 확보.
- 비밀번호 유출 위험이 Bitwarden 볼트 하나로 국한되고, 2FA로 이중 방어.
- 법인 전환 시 계정 승계 리스크 감소.

### 감수하는 것

- Apple Developer 등 일부 서비스는 원천적으로 개인 명의로 시작해야 하며, 전환 시점에 일회성 비용 발생.
- 공용 Gmail 비밀번호가 사실상 "팀 진입 루트 키" — 유출 시 연쇄 피해가 크므로 특별 관리 필요(2FA 필수, 변경 즉시 팀 알림).
- 무료 플랜 한도(멤버 2명, Collection 2개) 초과 시 Teams 플랜(월 $4/user)으로 전환 필요. 3번째 멤버 합류가 실질 트리거.

### 재검토 조건

- 3번째 멤버 합류 확정 시 Bitwarden Teams 플랜 전환 여부 결정. 비용·편의 트레이드오프 검토 후 본 ADR 확장 또는 새 ADR 작성.
- TOTP 공유가 실제 운영에서 병목이 되면 Bitwarden Premium(연 $10/user) 전환 검토.
- Apple Developer가 법인 Organization으로 전환되는 시점에 "공용 계정 정책의 적용 범위"를 Apple Developer 영역까지 확장하는 후속 ADR 작성.
