# ADR-002: 앱별 기술 스택 유연성 정책

- 작성일: 2026-04-22
- 작성자: 장수지, 박성훈
- 상태: accepted

## 배경

엔디즈는 매주 앱 1개씩 출시하는 다(多)앱 포트폴리오 모델을 운영한다. 앱별로 특성·규모·요구사항이 크게 다르다.

- 어떤 앱은 단순한 유틸리티(설정 1화면, 타이머 기능 등) — UIKit + UserDefaults로 충분
- 어떤 앱은 실시간 데이터·복잡한 상태 관리 필요 — SwiftUI + TCA 유리
- 어떤 앱은 다크모드 전제, 어떤 앱은 라이트모드만
- iOS 최소 지원 버전도 앱의 타깃 사용자에 따라 다름
- 서버도 앱마다 상황이 다름. 순수 로컬 앱부터 실시간 동기화가 핵심인 앱까지 범위가 넓다.

조직 차원에서 "공통 스택 강제"와 "앱별 자율 선택" 사이에서 결정이 필요했다.

## 결정

**앱마다 기술 스택을 자율 선택하는 정책을 채택한다.**

각 앱 레포는 아래 항목을 앱의 특성에 맞춰 독립적으로 결정한다:

- UI 프레임워크: SwiftUI / UIKit / 혼합
- iOS Deployment Target
- 아키텍처 패턴: MV / MVVM / TCA / Clean 등
- 데이터 저장소(로컬): SwiftData / CoreData / UserDefaults / Realm
- 다크모드 지원 여부
- 의존성 관리: SPM / CocoaPods
- 백엔드 방식: Firebase / Supabase / 서버리스 함수 / 직접 구축
- 데이터 저장소(서버): Firestore / Supabase Postgres / Realtime DB 등
- API 스타일: REST / GraphQL / SDK 직접 호출

다만 조직 차원의 **최소 공통 권장** 목록은 유지한다 (강제 아님):

- **Firebase Analytics + Crashlytics**: 모든 앱에서 기본 활성화 권장 (운영 가시성)
- **Swift Logger(OSLog) 기반 로깅**: 디버깅·DebugView 연계
- **SwiftLint**: 코드 스타일 일관성
- **BaaS 우선 검토**: 서버 전담 인력이 없으므로 앱 킥오프 시 Firebase/Supabase 등 BaaS를 먼저 고려한다. 커스텀 백엔드는 실제 요구가 검증된 후.

각 앱 레포의 README 또는 자체 ADR에 **기술 선택의 근거**를 기록한다.

## 근거

### 앱별 자율을 선택한 이유

- **과잉 표준화의 비용이 더 크다** — 매주 새 앱을 내는 모델에서, 유틸성 앱에 TCA를 강제하면 개발 시간이 2~3배로 늘어나 주간 출시 주기가 무너진다.
- **기술 탐색·학습 자산 축적** — 앱마다 다른 스택을 시도해볼 수 있어 팀의 기술 스펙트럼이 넓어진다. 면접·포트폴리오 어필에도 유리.
- **앱 수명에 따른 적정 기술** — 단기 실험 앱은 빠른 개발, 장수 앱은 유지보수성 중심으로 자연스럽게 선택이 갈린다.

### 최소 공통 권장만 유지하는 이유

- Analytics·Crashlytics가 없는 앱은 운영 중 장애를 감지할 수 없어 최소한 이건 통일해야 한다.
- 로깅·SwiftLint는 개발자 교대 시 진입 비용을 줄이는 정도의 영향이라 "권장"까지만.
- BaaS 우선 원칙은 현재 팀 구성(iOS 2명, 서버 전담 없음) 반영. 커스텀 서버는 운영 부담이 실제 개발 시간을 잠식하기 때문.

### 폐기한 대안

- **완전 공통 스택 강제** (예: 모두 SwiftUI + MVVM + Firebase): 일관성·온보딩 비용은 줄지만 앱 특성 무시로 개발 효율 손실. 2명 팀에서는 효율 손실이 더 크리티컬.
- **매월 스택 재선정 회의**: 오버헤드가 주간 출시 리듬을 깬다. 선택은 앱 킥오프 시점에 결정하고 그 앱 기간 내 변경 자제.

## 결과

### 얻는 것

- 앱 특성에 맞는 최적 스택으로 개발 효율 극대화.
- 팀이 다양한 기술을 실전에서 경험 → 기술 역량·포트폴리오 다각화.
- 앱별 README/ADR에 "왜 이 스택을 선택했는가" 기록이 쌓여 대외 설명 자산이 된다.

### 감수하는 것

- 멤버가 다른 앱을 지원할 때 초기 진입 비용 존재.
- "이 앱은 TCA라 어렵네" 같은 내부 격차 가능 — 문서화와 페어 코딩으로 보완.

### 재검토 조건

- 동일 스택 반복 선택이 5개 앱 이상 연속되면 사실상 공통 스택이 형성된 것 → 그 시점에 공통화 ADR 추가 검토.
- 서로 다른 스택 간 지원 전환 비용이 출시 일정을 2회 이상 지연시키면 공통화 재평가.
- 서버 관련 결정이 3개 이상 누적되면 별도 ADR(백엔드 전략)로 분리 검토.
