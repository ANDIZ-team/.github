# ADR-001: Git Flow 풀버전 채택

- 작성일: 2026-04-22
- 작성자: 장수지, 박성훈
- 상태: accepted

## 배경

엔디즈는 매주 앱 1개씩 출시·관리하는 다(多)앱 포트폴리오 운영 모델을 채택했다. 이 모델에서 브랜치 전략을 결정할 필요가 있었다.

주요 제약:
- 매주 앱 출시 + 여러 앱 동시 운영이라, 한 앱이 App Store 심사 대기 중에도 다음 기능 개발이 동시에 진행되어야 한다.
- 심사 거절 시 release 브랜치에서 격리된 수정이 가능해야 한다.
- 출시용 커밋(버전 bump, 인증서 교체 등)이 develop 히스토리에 섞여 들어오면 혼동이 생긴다.
- 2명 규모의 팀이므로 너무 무거운 전략은 리뷰·머지 부담이 과해진다.

## 결정

**Git Flow 풀버전을 채택한다.** 브랜치 구성:

- `main`: 출시된 코드만. 태그로 버전 관리.
- `develop`: 다음 출시 대상 통합 브랜치.
- `feature/*`: 기능 개발. develop에서 파생, develop으로 머지.
- `release/*`: 출시 준비. develop에서 파생, 안정화 후 main과 develop 양쪽에 머지.
- `hotfix/*`: 출시본 긴급 수정. main에서 파생, main과 develop 양쪽에 머지.

이 규약은 엔디즈 모든 앱 레포에 공통 적용한다. `.github` 메타 레포 등 문서 전용 레포는 예외.

## 근거

### Git Flow를 선택한 이유

- **심사 대기 중 develop 동결 없이 다음 기능 개발** — release 브랜치가 있어야 "심사 대기 중인 코드"와 "다음 주 개발 중인 코드"가 분리된다. Trunk-based는 이 분리가 없어 출시 중 개발이 멈추거나 feature flag로 우회해야 한다.
- **심사 거절 대응 격리** — release 브랜치에서 수정하고 다시 올릴 때 develop의 최신 변경이 섞이지 않는다.
- **출시 메타 커밋 분리** — 버전 bump, 인증서 교체, 메타데이터 수정이 main 히스토리에만 남아 릴리스 노트 추적이 쉽다.
- **주간 출시 사이클과 자연스럽게 매핑** — 주 초반: feature 개발, 주 중반: release 브랜치 컷, 주 후반: main 머지·태그 및 출시.

### 폐기한 대안

- **Trunk-based Development**: 2명 팀엔 리뷰 오버헤드가 적어 매력적이지만, 매주 심사가 끼는 모델에서 "develop 동결 vs 다음 기능 진행" 딜레마를 피할 수 없어 포기.
- **GitHub Flow (main + feature만)**: Trunk-based보다 단순하나 release 격리 자체가 없어 심사 거절 시 롤백·재수정이 develop를 오염시킨다.

## 결과

### 얻는 것

- 여러 앱을 동시에 각자 다른 출시 단계(개발/release/심사 대기)에 둘 수 있다.
- 심사 이슈가 생겨도 develop의 개발 속도가 멈추지 않는다.
- 릴리스 히스토리가 main 브랜치에 깨끗이 남아 대외 어필(면접, 포트폴리오) 자산이 된다.

### 감수하는 것

- release/hotfix 머지 시 main + develop **양쪽 머지 필수** (누락 시 다음 버전에 동일 버그 재등장).
- 브랜치 관리 규칙이 많아 2명 팀엔 초기 학습 비용이 있다.

### 재검토 조건

- release/hotfix 양쪽 머지 누락으로 실제 버그 재발 사례가 3회 이상 누적되면 자동화(GitHub Actions로 release/hotfix → main 머지 시 develop 자동 PR 생성) 도입을 우선 검토한다. 이 자동화로도 개선 안 되면 전략 재검토.
- 팀이 4명 이상으로 확장되고 레포 수가 10개를 넘으면 Trunk-based + feature flag 전환을 재평가한다.
